<domain>
## Phase Boundary

Minimal app activity that handles mic permission, binds to DictationService, and provides a test surface for recording/waveform/haptics features built in Phase 2. Also establishes test infrastructure (Compose UI tests, test doubles). This is a temporary dev-facing screen replaced by onboarding in Phase 4.

</domain>

<decisions>
## Implementation Decisions

### App home screen
- Minimal test surface: single screen with two sections — IME status card and recording test area
- Recording test area mirrors the keyboard recording UI: real-time waveform, timer, state display (idle/recording)
- IME status card shows whether Dictus keyboard is enabled/selected, with a button to open system keyboard settings if not enabled
- Branded but simple visual style: use DictusTheme colors and Material 3 components (cards, buttons), no logo/branding header
- This screen is replaced by onboarding flow in Phase 4

### Permission handling
- Just-in-time permission request: RECORD_AUDIO requested when user taps "Test Recording" button (not on app launch)
- Bundle POST_NOTIFICATIONS permission with RECORD_AUDIO request (required for foreground service notification on Android 13+)
- On denial: show Snackbar explaining why mic is needed + action button to open app settings
- Permission handling in this phase is app-side only — keyboard (IME) side permission handling deferred to Phase 4 onboarding

### Navigation structure
- Single screen, no Jetpack Navigation Compose
- YAGNI: add Navigation Compose in Phase 4 when onboarding requires multi-screen flow
- No bottom nav, no drawer, no tabs

### Test infrastructure
- Add Compose UI test dependencies across modules
- Write unit tests for new code (permission logic, service binding state)
- Write Compose UI tests for the test surface screen
- Create FakeDictationController in test sources (implements existing DictationController interface from core/) — emits controllable states for both app and ime module tests
- Add a root-level Gradle task alias for running all tests across modules
- 8 existing tests across 3 modules serve as foundation

### Claude's Discretion
- Exact Compose UI test library choices (compose-ui-test-junit4 vs alternatives)
- Service binding lifecycle management in MainActivity
- Exact layout of IME status card and recording test area
- Snackbar duration and styling
- FakeDictationController implementation details
- Gradle task naming convention

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `DictationController` interface (core/): Already abstracts service control — app test surface can bind through this same interface
- `DictationState` sealed class (core/): State model ready to drive test surface UI
- `WaveformBars.kt` (ime/): Existing waveform composable — can potentially be extracted to core/ or reused
- `DictusTheme` (core/): Branded theme ready for app screen styling
- `RecordingScreen.kt` (ime/): Recording UI reference for mirroring in app test surface

### Established Patterns
- Hilt DI: `@AndroidEntryPoint` for Activity, `EntryPointAccessors` for IME
- Compose-based UI rendering
- Timber structured logging throughout
- StateFlow for reactive state

### Integration Points
- `MainActivity` → binds to `DictationService` via `DictationController` interface
- `DictationService` exposes `DictationController` through its `LocalBinder`
- Test surface observes `DictationController.state` StateFlow for real-time UI updates
- Permission result feeds back into recording button enabled state

</code_context>

<specifics>
## Specific Ideas

- Mirror the keyboard recording UI (waveform, timer, state) in the app test surface so recording can be tested without switching apps
- IME status card replaces the current placeholder "Enable Dictus Keyboard in Settings" text with a proper status check + action button
- Phase 02.1 was inserted because you "cannot test recording/waveform/haptics without a minimal app activity that handles mic permission and service binding"
