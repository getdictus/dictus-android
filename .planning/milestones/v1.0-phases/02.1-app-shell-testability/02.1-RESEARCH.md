# Phase 02.1: App Shell + Testability - Research

**Researched:** 2026-03-24
**Domain:** Android Activity shell, Compose UI testing, runtime permissions, service binding
**Confidence:** HIGH

## Summary

Phase 02.1 builds a minimal app Activity that serves as a test surface for the recording/waveform/haptics features built in Phase 2. The existing `MainActivity` is a placeholder with static text -- it needs to become a functional screen that binds to `DictationService`, requests mic permission, and mirrors the keyboard recording UI (waveform, timer, state). A secondary goal is establishing Compose UI test infrastructure across all 3 modules.

The codebase already has the hard parts solved: `DictationController` interface in core, `DictationService` with `LocalBinder` in app, and `DictationState` sealed class driving UI. The IME already demonstrates service binding and state-driven recording UI switching. The app shell mirrors this pattern but with standard Activity lifecycle instead of IME lifecycle.

**Primary recommendation:** Bind to DictationService in MainActivity using a lifecycle-aware ServiceConnection, request RECORD_AUDIO + POST_NOTIFICATIONS via `RequestMultiplePermissions` contract, add Compose UI test dependencies with Robolectric for fast JVM-based tests, and create a FakeDictationController for test doubles.

### Pattern 1: Service Binding in Activity with Lifecycle Awareness
**What:** Bind to DictationService in onCreate, unbind in onDestroy, expose controller via StateFlow.
**When to use:** Activity needs to observe and control the DictationService.
**Example:**
```kotlin
// In MainActivity - mirrors the pattern already used in DictusImeService
@AndroidEntryPoint
class MainActivity : ComponentActivity() {

    private var dictationController: DictationController? = null
    private var isBound = false
    private val _controllerState = MutableStateFlow<DictationController?>(null)

    private val serviceConnection = object : ServiceConnection {
        override fun onServiceConnected(name: ComponentName?, binder: IBinder?) {
            val service = (binder as? DictationService.LocalBinder)?.getService()
            dictationController = service
            _controllerState.value = service
            isBound = true
        }
        override fun onServiceDisconnected(name: ComponentName?) {
            dictationController = null
            _controllerState.value = null
            isBound = false
        }
    }

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        bindDictationService()
        setContent {
            DictusTheme {
                val controller by _controllerState.collectAsState()
                TestSurfaceScreen(controller = controller)
            }
        }
    }

    override fun onDestroy() {
        if (isBound) {
            unbindService(serviceConnection)
            isBound = false
        }
        super.onDestroy()
    }

    private fun bindDictationService() {
        val intent = Intent(this, DictationService::class.java)
        bindService(intent, serviceConnection, Context.BIND_AUTO_CREATE)
    }
}
```

**Key difference from IME binding:** MainActivity can import DictationService directly (same module) and cast LocalBinder directly, unlike the IME which uses reflection to avoid circular dependencies.

### Pattern 2: IME Status Check
**What:** Check if Dictus keyboard is enabled in system settings and if it's the currently selected IME.
**When to use:** IME status card needs to show status and provide action to enable.
**Example:**
```kotlin
// Two-level check: enabled (in system list) vs selected (currently active)
fun isImeEnabled(context: Context): Boolean {
    val imm = context.getSystemService(Context.INPUT_METHOD_SERVICE) as InputMethodManager
    return imm.enabledInputMethodList.any {
        it.packageName == context.packageName
    }
}

fun isImeSelected(context: Context): Boolean {
    val currentIme = Settings.Secure.getString(
        context.contentResolver,
        Settings.Secure.DEFAULT_INPUT_METHOD
    )
    return currentIme?.startsWith(context.packageName) == true
}

// Open system keyboard settings
fun openKeyboardSettings(context: Context) {
    context.startActivity(Intent(Settings.ACTION_INPUT_METHOD_SETTINGS))
}
```

### Pattern 3: Multiple Permission Request with Snackbar Denial
**What:** Request RECORD_AUDIO + POST_NOTIFICATIONS together, show Snackbar on denial.
**When to use:** User taps "Test Recording" button.
**Example:**
```kotlin
@Composable
fun PermissionAwareRecordingButton(
    onPermissionGranted: () -> Unit,
) {
    val context = LocalContext.current
    val snackbarHostState = remember { SnackbarHostState() }
    val scope = rememberCoroutineScope()

    val permissionLauncher = rememberLauncherForActivityResult(
        ActivityResultContracts.RequestMultiplePermissions()
    ) { permissions ->
        val audioGranted = permissions[Manifest.permission.RECORD_AUDIO] == true
        if (audioGranted) {
            onPermissionGranted()
        } else {
            scope.launch {
                val result = snackbarHostState.showSnackbar(
                    message = "Microphone permission is required for dictation",
                    actionLabel = "Settings",
                    duration = SnackbarDuration.Long,
                )
                if (result == SnackbarResult.ActionPerformed) {
                    // Open app settings
                    val intent = Intent(Settings.ACTION_APPLICATION_DETAILS_SETTINGS).apply {
                        data = Uri.fromParts("package", context.packageName, null)
                    }
                    context.startActivity(intent)
                }
            }
        }
    }

    // Build permission list based on API level
    val permissionsToRequest = buildList {
        add(Manifest.permission.RECORD_AUDIO)
        if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU) {
            add(Manifest.permission.POST_NOTIFICATIONS)
        }
    }.toTypedArray()

    Button(onClick = {
        permissionLauncher.launch(permissionsToRequest)
    }) {
        Text("Test Recording")
    }
}
```

### Pattern 4: FakeDictationController for Testing
**What:** Test double that implements DictationController with controllable state emission.
**When to use:** UI tests and unit tests that need to drive recording state without real audio.
**Example:**
```kotlin
// In core/src/test/ (shared across modules)
class FakeDictationController : DictationController {
    private val _state = MutableStateFlow<DictationState>(DictationState.Idle)
    override val state: StateFlow<DictationState> = _state.asStateFlow()

    var startRecordingCallCount = 0
        private set
    var stopRecordingCallCount = 0
        private set
    var cancelRecordingCallCount = 0
        private set
    var stopRecordingResult = FloatArray(0)

    override fun startRecording() {
        startRecordingCallCount++
        _state.value = DictationState.Recording(elapsedMs = 0L, energy = emptyList())
    }

    override fun stopRecording(): FloatArray {
        stopRecordingCallCount++
        _state.value = DictationState.Idle
        return stopRecordingResult
    }

    override fun cancelRecording() {
        cancelRecordingCallCount++
        _state.value = DictationState.Idle
    }

    // Test helpers
    fun emitState(state: DictationState) {
        _state.value = state
    }
}
```

### Anti-Patterns to Avoid
- **Binding in Composable via DisposableEffect:** Service binding belongs in Activity lifecycle, not Compose composition. Configuration changes cause recomposition which would trigger unbind/rebind unnecessarily.
- **Checking permissions in composable body:** Permission checks must happen in callbacks (onClick), not during composition. Side effects during composition cause infinite recomposition loops.
- **Sharing WaveformBars via copy-paste from ime:** If the composable is reused, extract to core module. If the app test surface only approximates it, a simpler version is fine.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Permission request flow | Custom dialog + ActivityCompat.requestPermissions | `rememberLauncherForActivityResult(RequestMultiplePermissions())` | Handles lifecycle, rotation, result callbacks automatically |
| IME enabled check | Parsing Settings.Secure strings manually | `InputMethodManager.getEnabledInputMethodList()` | System API handles all edge cases, returns structured data |
| Compose test rule | Manual ComposeView setup in tests | `createComposeRule()` + Robolectric | Handles Compose test clock, idle synchronization, assertions |
| Snackbar with action | Custom Toast or Dialog | `SnackbarHostState.showSnackbar()` with actionLabel | Material 3 standard, handles duration/dismissal/action |

**Key insight:** The permission + service binding patterns are well-established Android patterns. The only novel part is the IME status check, which is a simple 2-line API call.

## Common Pitfalls

### Pitfall 1: POST_NOTIFICATIONS only exists on API 33+
**What goes wrong:** App crashes on API 29-32 devices if you request POST_NOTIFICATIONS unconditionally.
**Why it happens:** POST_NOTIFICATIONS was added in Android 13 (API 33). On earlier versions, the permission doesn't exist.
**How to avoid:** Guard with `Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU` before adding to the permission array. On API < 33, notifications are granted by default.
**Warning signs:** Crash on Pixel 4 running Android 10-12.

### Pitfall 2: Service binding race condition
**What goes wrong:** User taps "Test Recording" before ServiceConnection.onServiceConnected fires.
**Why it happens:** `bindService()` is async. There's a window between calling bind and receiving the callback.
**How to avoid:** Disable the recording button when `controller == null`. Use the `_controllerState` StateFlow to drive button enabled state.
**Warning signs:** NullPointerException on dictationController access.

### Pitfall 3: Robolectric test configuration
**What goes wrong:** Robolectric tests fail with SDK version mismatch or missing shadows.
**Why it happens:** Robolectric needs to be told which SDK to emulate, and some Android APIs aren't shadowed.
**How to avoid:** Add `@Config(sdk = [35])` or a `robolectric.properties` file with `sdk=35`. Use `testOptions.unitTests.isIncludeAndroidResources = true` in build.gradle.kts.
**Warning signs:** `android.content.res.Resources$NotFoundException` in test output.

### Pitfall 4: Compose UI test manifest missing
**What goes wrong:** `createComposeRule()` fails with "No activity found" error.
**Why it happens:** The test rule needs a ComponentActivity to host the Compose content. `compose-ui-test-manifest` provides this.
**How to avoid:** Add `debugImplementation(libs.compose.ui.test.manifest)` to the module.
**Warning signs:** `ActivityNotFoundException` in test logs.

### Pitfall 5: shouldShowRequestPermissionRationale not available in Compose
**What goes wrong:** Developer tries to show rationale before permission request but can't access the Activity method from a composable.
**Why it happens:** `shouldShowRequestPermissionRationale` is an Activity method, not available in composable scope.
**How to avoid:** For this phase, keep it simple: always request, show Snackbar on denial. The Snackbar with "Settings" action handles the permanently-denied case too.
**Warning signs:** Trying to call Activity methods from composable functions.

// Source: Android InputMethodManager API
import android.provider.Settings
import android.view.inputmethod.InputMethodManager

data class ImeStatus(
    val isEnabled: Boolean,  // Listed in system keyboard settings
    val isSelected: Boolean, // Currently active keyboard

fun getImeStatus(context: Context): ImeStatus {

        it.packageName == context.packageName

    val currentIme = Settings.Secure.getString(

        Settings.Secure.DEFAULT_INPUT_METHOD
    )
    val isSelected = currentIme?.startsWith(context.packageName) == true
    return ImeStatus(isEnabled = isEnabled, isSelected = isSelected)
}

## Open Questions

1. **WaveformBars reuse strategy**
   - What we know: `WaveformBars` composable lives in `ime/src/main/`. The app test surface needs a similar waveform.
   - What's unclear: Whether to extract WaveformBars to core/ or create a simpler version in app/.
   - Recommendation: Extract to core/ since both app and ime need it. This is a small, pure composable with no ime-specific dependencies (only uses DictusColors from core).

2. **FakeDictationController location**
   - What we know: CONTEXT.md says "Create FakeDictationController in test sources". Both app and ime modules need it.
   - What's unclear: Putting it in `core/src/test/` means it's only available to core's tests. Other modules would need their own copy or a shared test fixtures approach.
   - Recommendation: Place in `core/src/testFixtures/` using Gradle's `java-test-fixtures` plugin, or place in `core/src/test/` and duplicate in app/ime test sources. The simpler approach (duplicate) is fine for a ~30 line class.
